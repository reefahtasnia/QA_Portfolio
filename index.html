01 / The Program

A training ground, not a job title.

I completed this training program over two months. It started with group mentorship, working through the fundamentals of manual testing alongside other trainees: finding bugs, learning to write them up properly, and understanding what actually counts as a defect.

That was followed by one to one mentorship with an experienced QA mentor, working through full applications and producing complete test documentation rather than just bug lists.

What follows isn't a certificate. It's a sanitized look at the actual work: how I approached testing, and what the defect reports and documentation looked like by the end of it.

Group Mentorship

A few weeks of foundational training alongside other trainees. Functional testing on small training applications, and learning the anatomy of a proper defect report.

One to One Mentorship

A few weeks of individual training with an experienced mentor. Full test documentation suites: acceptance sheets, test surveys, and quality reports, reviewed and corrected directly.

02 / My Strategy

How I actually test something.

This is the process I run for manual functional and GUI testing. Specifics below are described by category, not reproduced verbatim. This is methodology, not a copy of real defect text.

The Pass, In Order

Six tiles, hover one to read it, or press play to walk through the whole sequence.

01
Read everything first

Full spec, every page, before the application is even open. This single habit prevents more missed defects than any technique that follows it.

02
Decompose the app

Break it into modules by logic, by role, or by navigation. A bug is only easy to triage if it's easy to locate first.

03
Pick the documentation

Match the format to the project. A small stable feature gets a checklist. A complex, customer-facing flow earns a full test survey.

04
Design before clicking

Equivalence classes and boundary values are chosen on paper first, so execution is deliberate instead of exploratory guesswork.

05
Execute, element by element

Every interactive element gets its own pass. Broad sweeps miss things that individual attention catches.

06
Report, then verify

A defect isn't done at submission. Severity gets graded on impact, and the report gets checked against the spec one more time before it ships.

How I Decide Things

Functional
Is this in scope?

I check the spec and verify with a business analyst before logging anything I can't yet prove is a defect rather than intended behavior. Stub buttons and by-design quirks get confirmed, not assumed.

Functional
What's the actual severity?

I grade impact on function and experience, never urgency. A broken core flow outranks a cosmetic flaw regardless of how visible either one is.

GUI
Is this one issue or several?

Same defect across multiple places stays one report with the other locations noted. Different defects found in the same minute never get merged just because they were found together.

GUI
What earns a screenshot vs. a video?

A static, single-state issue gets a still image with the problem marked. Anything involving a transition or multi-step interaction gets a short recording instead.

Functional
How many values actually need testing?

Not every value, every class. I pick one representative input per equivalence class, then test hard at the boundaries, since that's where defects cluster.

GUI
Does the wording hold up?

If a title needs a hedge word to make sense, it isn't ready yet. Every claim in a report needs to be provable from what was directly observed.

What This Looks Like in Practice

Situation

During a structured pass through a checkout flow, the displayed order total and the amount actually charged on submission didn't match.

Severity call

Graded as a high-impact functional defect. It touches a core transaction path and affects every user who completes that flow, not an edge case.

Scope check

Verified against the specification first to confirm the displayed total was the source of truth, rather than assuming it without checking.

What I didn't do

Set a priority, group it with an unrelated layout issue found minutes later, or describe it with language stronger than what was directly observed.

Group Mentorship Phase

The Spec-First Lesson

40 bugs filed

My first full testing pass on a small ordering application. I started testing before finishing the specification and missed defects that were sitting in plain text on a page I hadn't read yet.

40Defects logged
Bug reportsDocumentation level
FunctionalPrimary test type
What I missed by skipping the spec
  • A required selection step in the ordering flow that wasn't present
  • An authentication step the spec called for that the build didn't have
  • A date-restriction rule that wasn't being enforced
  • A timestamp that should have come from the server rather than the local device
What changed in my process
  • Full spec read, end to end, before opening the browser
  • Source code scanned for logic before runtime clicks
  • Every answer from a business analyst cross-checked against the code itself
  • Now standard on every task since

Major Category: Functional

A quantity control allowed its value to go negative instead of stopping at its lower limit.

What I did

Decremented the control repeatedly past its expected floor value.

Expected

The control should stop decrementing once it reaches its minimum valid value.

Observed: the value continued past zero into negative numbers, and dependent quantities recalculated incorrectly as a result.
